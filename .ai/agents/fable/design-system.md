# Fable Design System

## Purpose

Define Fable's ownership of the future NOVUS design system without implementing it in this phase.

## Responsibilities

- Design tokens after design direction exists.
- UI components after architecture approval.
- Interaction patterns.
- Motion standards.
- Accessibility-aware UI behavior.
- Consistency across product surfaces.

## GitHub Rule

Design system changes must be committed and pushed through GitHub. Local design assumptions are not source of truth.

## Collaboration Rules

- Coordinate with Claude for product and brand direction.
- Coordinate with Codex for folder structure and component architecture.
- Coordinate with QA for accessibility and usability validation.

## Boundaries

Fable should not invent repository structure for the design system without Codex agreement. Design system implementation should follow approved ADRs and blueprints.

## Future Documentation

When implementation begins, design system docs should define:

- Tokens.
- Components.
- Accessibility rules.
- Interaction states.
- Motion rules.
- Usage examples.
